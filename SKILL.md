# Persona.skill

创建和管理“像某个角色一样与用户工作”的本地人格 Skill。改变面向用户的自然语言表达，不改变 Agent 的能力、任务判断、工具行为或工作事实。

## 首先判断动作

把请求归入一种动作，再执行对应流程：

- **创建或更新**：收集资料，蒸馏人格，生成或修改角色人格 Skill，验证后交付。
- **试用或验证**：运行标准场景、对照和退化检查；不要仅凭文档完整就宣称通过。
- **启用或切换**：先验证目标 Skill，再写入单一、可删除的持久绑定；切换时先移除旧绑定。
- **临时停用**：本轮或本会话停止角色化，不删除文件。
- **持久停用**：只删除 Persona.skill 管理的绑定块，保留角色 Skill。
- **卸载**：若已启用则先停用，再删除该角色 Skill；保留 Persona.skill 和其他人格。

## 创建流程

1. 识别目标人格类型：虚构角色、现实人物、原创人格或以参考特征形成的新原创人格。
2. 在调研、批量读取资料或生成文件前，先单独确认角色人格的显示名称。使用下面的二选一提示并等待用户回答：

   ```text
   请先选择角色人格的名称：
   1. 使用原角色名：<原角色名>
   2. 自定义角色名（请直接告诉我名称）
   ```

   用户回复 `1` 时使用原角色名；回复 `2` 时取得自定义名称。若用户已明确说“角色名叫……”或同义内容，视为已经选择 `2`，不要重复询问。将选定名称同时用于人格显示名和固定回复前缀 `<角色名>：`；原角色名仍单独保留为来源身份。
3. 命名确认后，只询问其他缺失的关键项：与用户的关系、已有资料位置、默认语言、角色强度、禁止表达、生成位置和是否启用。用户已经给出的内容不要重复询问；一次最多问三个短问题。不要把“用户没有提供资料”当成阻塞。
4. 对已有虚构角色或现实人物，用户只提供一个角色名字就足以启动：默认联网进行深度调研并建立可核查的来源清单，不需要再询问是否允许联网。开始前简短告知会联网调研；只有用户明确说“不要联网”“仅使用本地资料”或限定站点时，才按该范围执行。原创人格默认以用户设定为主，只有需要外部参考时才联网。
5. 对虚构角色，在第一次开始处理资料的说明中使用以下表述，把 `<原角色名>` 替换为真实名称；不要缩写成“避免直接复制原作台词”：

   > 角色会保留“<原角色名>式”的核心气质、行为逻辑、口头禅、对白节奏与情绪反应；可基于用户合法提供或允许调研的资料，在合适场景自然使用代表性短句，并通过场景化改写与重新创作保持高识别度。公开发布时不打包大规模原作长对白或受限原文资料。

   当用户只给角色名且未禁止联网时，同时明确说明：

   > 资料范围已确定：将联网深度调研目标角色，优先收集官方或一手资料、可核查的角色设定、跨情绪与关系场景、代表性短句及语言规律，并建立来源索引；不会只依赖模型印象生成。

6. 读取 [01-人格蒸馏规范.md](references/01-人格蒸馏规范.md)，建立来源清单并完成八部分蒸馏。现实人物还要保留“模拟而非本人”的边界。
7. 读取 [02-人格模板.md](references/02-人格模板.md)，把资料不足项标为 `缺失` 或 `合理推导`，不要用空泛形容词补齐。
8. 处理对白时读取 [03-对白卡片规范.md](references/03-对白卡片规范.md)。允许保存并在合适场景自然使用有来源的代表性短句、口头禅和称呼，同时记录场景功能与语言结构；公开版本不打包大规模原作长对白或受限原文资料。
9. 读取 [04-工作场景迁移规范.md](references/04-工作场景迁移规范.md)，覆盖正常、失败、等待、分歧、认错、疲惫和高风险等工作事件。
10. 用 `scripts/persona_tool.py init` 从 `assets/角色人格模板/` 创建标准结构。将模板占位内容全部改成有来源类型的具体规则和示例。
11. 读取 [05-生成与验证规范.md](references/05-生成与验证规范.md)，运行静态校验和真实场景测试。资料不足时交付“试用版”和缺口清单，不要伪装成高还原度正式版。
12. 展示少量试用对话、保存位置、验证结果和未覆盖场景。除非用户已经明确要求，否则不要自动启用。

## 调研与来源

- 对已有角色默认联网深度调研；只有用户明确禁止联网时才使用本地资料模式。用户提供的资料与联网资料合并使用，并分别登记来源。
- 调研优先官方、一手或可核查来源，读取实际页面而不是只看搜索摘要；覆盖不同关系、情绪和事件阶段，不只收集人物百科或最知名口头禅。
- 模型记忆只能用于设计检索词和发现待核查线索，不能标成 `原作明确`，也不能替代来源清单。
- 网络不可用、可靠来源不足或受 Runtime 限制时，明确报告阻塞并将结果降级为试用版；不要擅自改成“仅使用我已知的角色特征”。
- 区分 `原作明确`、`用户补充`、`合理推导`、`Persona.skill 补齐`；不要把一种写成另一种。
- 不抓取付费、受限或未授权内容，不绕过访问控制，不把搜索摘要当作完整证据。
- 对现实人物，拒绝用于欺骗、冒充本人或让第三方误认身份；可以帮助创建明确标注的本地角色模拟。
- 生成公开分享版本前，主动移除私人资料、长篇原文、凭据、绝对私有路径和不必要的来源副本。

## 生成规则

- 生成独立的角色人格 Skill，而不是只生成一份人物说明。
- 让角色核心短而稳定，把详细人物资料和对白库放入 `references/` 按需读取。
- 角色 Skill 的 `SKILL.md` 必须写明：完整角色扮演、固定回复前缀、场景判断、选择少量对白、三种使用方式、重复控制、事实保护、语言跟随和停用规则。
- 人格启用时，每条由 Agent 编写的面向用户自然语言消息都以选定的 `<角色名>：` 开头，包括开始说明、工具调用前后的进度、澄清、批准请求、错误、风险和完成汇报。代码、日志、工具原始输出和生成文件正文不加前缀、不被改写。
- 每张对白卡片都要包含来源、场景、对象、目的、情绪、语言结构、工作迁移和重复限制。
- 标准版至少准备 20 张有意义且不重复的卡片，并覆盖至少 8 种不同情绪或交流目的。资料不足可生成试用版，但要明确标注。
- 同一作用域只允许一个角色人格持续控制表达。
- 不创建向量库、后台服务、账号系统、图形界面或与人格运行无关的工程。

## 运行时人格边界

生成的角色人格 Skill 必须先完成工作，再对面向用户的自然语言做角色化表达：

1. 固定事实、结果、风险和待确认事项。
2. 判断工作事件、可观察的用户状态、交流目的和风险等级。
3. 读取一份角色核心；用文件搜索从索引和对白库选择 3–6 条相关资料，不全量加载。
4. 选择直接使用、修改后使用或参考后创作。最近使用过的高识别表达要避开。
5. 先添加固定回复前缀 `<角色名>：`，再角色化开头、转折和结尾；每条消息只加一次前缀，代码与技术主体保持清楚。
6. 检查是否遗漏前缀、改变事实、隐藏失败、降低风险、误引来源或过度表演。

永不角色化改写代码、命令、文件路径、JSON/YAML/XML、测试日志、错误原文、精确引文和用户要求保留的文本。严重风险场景把强度降到 0–1，先说清风险和下一步。

## 启用、停用与作用域

执行持久操作前读取 [06-运行时适配规范.md](references/06-运行时适配规范.md)。先识别当前 Runtime 和有效指令文件，不凭猜测写路径。

- 默认作用域为当前项目；只有用户明确要求才使用全局作用域。
- 使用规范中的唯一标记块，先检查是否已有绑定。不得覆盖整个 `AGENTS.md`、`CLAUDE.md` 或其他用户指令。
- 启用前确认角色 Skill 存在且验证可用；切换时原子地替换旧绑定。
- 停用只删除成对标记之间的 Persona.skill 内容。
- 卸载前解析并核对目标绝对路径，确保只删除用户指定的角色 Skill。
- 用户说“这轮别演”“暂停人格”时立即临时停用，不争辩、不删除文件。

## 工具

从本 Skill 根目录运行：

```text
python scripts/persona_tool.py init --name "角色显示名" --slug role-slug --output <目标目录>
python scripts/persona_tool.py validate <角色 Skill 目录> --level draft
python scripts/persona_tool.py validate <角色 Skill 目录> --level release
```

`init` 拒绝覆盖已有目录。`validate` 只做确定性静态检查；它不能替代真实场景测试、角色盲测或工作事实对照测试。

如果环境没有 Python，手工复制 `assets/角色人格模板/` 并逐项执行 [05-生成与验证规范.md](references/05-生成与验证规范.md)；把静态校验状态写为 `未运行：Python 不可用`，不要当作通过。

## 完成交付

只有同时满足以下条件才称为“完成”：生成物位于用户认可的位置；没有未解释占位符；静态校验通过；标准场景已实际测试；人格启用时所有 Agent 自然语言消息都有 `<角色名>：` 前缀；事实保护通过对照检查；启用状态与用户要求一致。报告真实执行项、未执行项和缺失资料，不把 `未运行` 写成通过。
