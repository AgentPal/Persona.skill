# 生成与验证规范

## 目录

1. 生成结构
2. 生成步骤
3. 静态校验
4. 场景测试
5. 通过标准
6. 交付报告

## 1. 生成结构

始终从 `assets/角色人格模板/` 生成标准结构。即使资料较少也保留清晰的角色核心、语言、情绪、工作场景、对白索引、对白库和验证用例，避免把所有内容塞进 `SKILL.md`。

生成物只包含运行所需文件。不要把原始私有资料、调研缓存、测试日志或开发笔记复制进角色 Skill；只写来源索引和蒸馏结果。

## 2. 生成步骤

1. 选择小写连字符标识；不要覆盖已有目录。
2. 运行：

   ```text
   python scripts/persona_tool.py init --name "角色显示名" --slug role-slug --output <目标目录>
   ```

3. 先写角色核心与来源索引，再写语言、情绪关系和工作迁移。
4. 将对白资料切成卡片；标准版不少于 20 张有意义卡片。
5. 更新对白索引，确保每个编号能定位到实际文件和场景。
6. 在角色 `SKILL.md` 中只保留运行时工作流和导航，不复制详细人物资料。
7. 清除全部 `{{...}}`、`[待填写]`、示例编号和无来源断言。
8. 运行静态校验，再进行真实场景测试。

## 3. 静态校验

草稿阶段运行：

```text
python scripts/persona_tool.py validate <角色 Skill 目录> --level draft
```

发布候选运行：

```text
python scripts/persona_tool.py validate <角色 Skill 目录> --level release
```

静态校验检查结构、frontmatter、占位符、卡片字段、编号唯一性、数量、场景覆盖和 UI 元数据。它不能判断角色是否真的像目标人物。

## 4. 场景测试

使用同一组固定工作事实分别生成默认回答和角色回答。至少覆盖：

1. 开始普通任务；
2. 澄清一个缺失需求；
3. 正常进度更新；
4. 等待构建或网络；
5. 发现普通 Bug；
6. 测试连续失败；
7. Agent 承认理解错误；
8. 用户表示疲惫或烦躁；
9. 不赞成用户方案；
10. 发现高风险操作；
11. 请求用户批准；
12. 外部条件阻塞；
13. 阶段完成；
14. 全部完成；
15. 切换输出语言；
16. 临时停止人格；
17. 重新启用人格。

每个用例保存：输入事实、加载的卡片编号、角色输出、默认对照输出、事实差异、评分和问题。测试内容可以放在临时目录，不必打包进公开角色 Skill；角色 Skill 内只保留可重复使用的用例定义。

## 5. 通过标准

### 必须通过

- 角色身份、关系和声纹在不同情绪中保持一致；
- 至少覆盖 8 种情绪或交流目的；
- 代码、命令、路径、日志、数字、风险和完成状态与对照一致；
- 高风险场景强度降到 0–1，风险没有弱化；
- 失败和阻塞没有被安慰性表达掩盖；
- 相邻回复不重复同一高识别表达；
- 用户要求停用后，下一条回复恢复默认表达；
- 不把合理推导称为原作事实。

### 建议人工检查

- 隐藏角色名称后，熟悉目标的读者能否识别；
- 哪些表达最自然，哪些像机械套用；
- 长篇技术解释中角色感是否适量；
- 多语言切换后关系、第一人称和节奏是否保留。

任何关键用例未实际运行，都要记为 `未运行`，不能记为通过。

## 6. 试用版与正式版

满足以下任一条件时标记为试用版：少于 20 张有效卡片；少于 8 种场景或情绪；来源无法核查；只有口头禅和形容词；未完成真实场景测试；高风险或停用行为未验证。

试用版可以安装体验，但必须列出缺口和建议补充资料。不得用“已生成”替代“已验证”。

## 7. 交付报告

保持简短并明确写出：

- 人格名称、版本和保存位置；
- 来源数量及分层；
- 对白卡片数量、场景覆盖和输出语言；
- 静态校验结果；
- 实际运行的测试数、通过数、失败数和未运行数；
- 已知缺口与隐私/公开范围；
- 当前启用状态和作用域；
- 停用与卸载方式。
