# 生成与验证规范

## 1. 生成结构

始终从 `assets/角色人格模板/` 生成标准结构。即使资料较少也保留清晰的角色核心、语言、情绪、工作场景、对白索引、对白库和验证用例，避免把所有内容塞进 `SKILL.md`。

生成物分为私有本地资料层与可选的发布副本。完整对白、剧本、字幕或转写可保留在私有资料层，供切分、检索和再蒸馏；Persona.skill 不按版权类别、文本长度或资料完整度拒收。用户要求清理版时，发布副本只保留其确认的内容；不要擅自删除私有原件。

## 2. 生成步骤

1. 创建请求的第一条回复只能显示名称二选一：`1. 使用原角色名`；`2. 自定义角色名`，然后停止并等待。收到选择前不得调研、读资料或创建文件。把选择结果同时写入人格显示名和固定回复前缀 `<角色名>：`，把原角色名作为来源身份单独保留。
2. 名称确定后，创建过程中的每条 Agent 自然语言消息都从 `<角色名>：` 开始。对已有角色默认联网深度调研；即使用户只给角色名，也要建立来源索引并覆盖不同关系、情绪、事件阶段和代表性语言。
3. 选择小写连字符标识；不要覆盖已有目录。
4. 运行：

   ```text
   python scripts/persona_tool.py init --name "角色显示名" --slug role-slug --output <目标目录>
   ```

5. 默认联网深度调研；只有用户明确要求不联网时才限于本地。用户提供字幕、剧本、转写或其他有权使用的完整资料时，可建立私有资料层并登记来源、范围和不可公开标记。用户要求公开导出时，单独生成其确认范围内的发布副本。
6. 先写角色核心与来源索引，再写语言、情绪关系和工作迁移。
7. 将逐字对白资料切成原文卡。现有虚构角色正式版目标为至少 80 张彼此不重复且经来源核对的逐字原文卡、覆盖 40 个不同原作场景，其中至少 20 张为核心或常用表达；建议资料丰富时达到 120–300 张。场景摘要、人物分析、合理推导和工作改写不能进入对白库或计数。原创人格至少 20 张原创规范对白；现实人物模拟至少 40 张逐字原文卡。
8. 任一目标未达到时强制扩大调研范围，至少扩展站点、查询词、资料类型、版本、别名和语言。每轮把范围、结果和新增卡片数写入来源索引。不得在第一次检索不足后结束。
9. 扩大后达到目标，写 `调研状态：达标`；检查所有合理可访问范围后仍不足，写 `调研状态：已穷尽`、完整扩大范围和未达标原因。两种状态都生成正式版，不存在低版本。
10. 更新对白索引，确保每个编号能定位到实际文件和场景。
11. 在角色 `SKILL.md` 中只保留运行时工作流和导航，并明确所有 Agent 自然语言消息的固定回复前缀；不复制详细人物资料。
12. 清除全部 `{{...}}`、`[待填写]`、示例编号和无来源断言。
13. 运行正式版静态校验，再进行真实场景测试。通过后默认在当前会话启用；用户明确要求只创建时除外。

## 3. 静态校验

正式版校验运行：

```text
python scripts/persona_tool.py validate <角色 Skill 目录> --level release
```

静态校验检查结构、frontmatter、人格来源类型、正式版标记、调研状态、扩大范围记录、占位符、原文卡字段、编号和逐字原文唯一性、逐字原文卡数、不同原作场景数、核心/常用表达数、来源覆盖、工作场景覆盖、选择器和 UI 元数据。达到目标或具备完整 `已穷尽` 记录都可以通过正式版的数量覆盖检查；后者会保留覆盖警告。`已穷尽` 不能豁免缺失逐字原文、伪造来源或在对白库预写工作改写。

校验器不按版权类别、单条文本长度或私有原始资料总量拒绝本地资料库。版权内容可以参与私有蒸馏；访问控制和公开再分发范围由用户负责确认。

## 4. 场景测试

使用同一组固定工作事实分别生成默认回答和角色回答。至少覆盖：

1. 开始普通任务；
2. 澄清一个缺失需求；
3. 正常进度更新；
4. 等待构建或网络；
5. 发现普通 Bug；
6. 测试连续失败；
7. Agent 承认理解错误；
8. 用户表示疲惫或烦躁；
9. 不赞成用户方案；
10. 发现高风险操作；
11. 请求用户批准；
12. 外部条件阻塞；
13. 阶段完成；
14. 全部完成；
15. 切换输出语言；
16. 临时停止人格；
17. 重新启用人格。

每个用例保存：输入事实、加载的卡片编号、角色输出、默认对照输出、事实差异、前缀检查、评分和问题。必须覆盖开始说明、工具调用前后进度、澄清、风险和最终交付等不同消息类型。测试内容可以放在临时目录，不必打包进公开角色 Skill；角色 Skill 内只保留可重复使用的用例定义。

## 5. 通过标准

### 必须通过

- 角色身份、关系和声纹在不同情绪中保持一致；
- 人格启用时，每条由 Agent 编写的面向用户自然语言消息均从固定的 `<角色名>：` 开始，并且每条消息只出现一次前缀；
- 临时停用后不再添加角色名前缀，重新启用后恢复；
- 至少覆盖 8 种情绪或交流目的；
- 代码、命令、路径、日志、数字、风险和完成状态与对照一致；
- 高风险场景强度降到 0–1，风险没有弱化；
- 失败和阻塞没有被安慰性表达掩盖；
- 相邻回复不重复同一高识别表达；
- 用户要求停用后，下一条回复恢复默认表达；
- 不把合理推导、场景摘要或工作改写称为原文。
- 现有虚构角色达到 80 张逐字原文卡、40 个不同原作场景、20 张核心/常用原文卡、12 种情绪或交流目的、5 个来源条目且至少 3 个来源支持原作明确内容；或来源索引证明已强制扩大并穷尽合理可访问资料；无论走哪条路径，现有角色的对白库都只能保存逐字原文卡；
- 从大型资料库对失败、等待、危险、安慰和成功等查询运行选择器，每次只返回最匹配的 3–6 张卡片，并能排除最近已使用编号；

### 建议人工检查

- 隐藏角色名称后，熟悉目标的读者能否识别；
- 哪些表达最自然，哪些像机械套用；
- 长篇技术解释中角色感是否适量；
- 多语言切换后关系、第一人称和节奏是否保留。

任何关键用例未实际运行，都要记为 `未运行`，不能记为通过。

## 6. 正式版覆盖路径

最终只有正式版，资料覆盖有两条合法路径：

1. `达标`：逐字原文数量、不同原作场景、核心/常用表达、情绪目的和来源均达到目标。
2. `已穷尽`：未达到一个或多个目标，但已记录 `RESEARCH-01` 初始轮次和至少一个 `RESEARCH-02` 扩大轮次、检查过的站点/资料类型/版本/别名/语言、每轮结果及无法继续获得资料的原因。使用全部可得资料正式交付，并如实报告缺口。

不能用 `已穷尽` 掩盖单次搜索、只查官方页面、版权顾虑、没有完整剧本或不愿继续调研。版权内容可以进入用户确认的私有本地资料层；只有访问控制不得绕过。缺失逐字原文、用摘要或改写充数、重复原句凑数、来源冒充、缺失前缀、选择器失效或场景测试未执行都不能由 `已穷尽` 豁免。

## 7. 交付报告

保持简短并明确写出：

- 人格名称、版本和保存位置；
- 来源数量及分层；
- 对白卡片数量、场景覆盖和输出语言；
- 逐字原文卡数量、不同原作场景数量、核心/常用原文卡数量，以及被拒绝的非原文卡数量；
- 角色显示名与固定回复前缀；
- 静态校验结果；
- 实际运行的测试数、通过数、失败数和未运行数；
- 已知缺口与隐私/公开范围；
- 调研状态：`达标` 或 `已穷尽`，以及扩大范围记录；
- 当前会话启用状态和持久作用域；
- 停用与卸载方式。
