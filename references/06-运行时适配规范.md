# 运行时适配规范

## 1. 原则

安装、发现和持久指令位置可能随 Runtime 变化。先检查当前环境、用户指定范围和已有配置；只使用已确认的机制。角色 Skill 的内容保持 Agent Skills 兼容，Runtime 差异只放在安装和绑定层。

创建完成后默认立即在当前会话启用，因此完成报告和后续回复都使用角色名前缀；能安全确定 Runtime 当前项目指令文件时，同时持久绑定当前项目。用户明确要求“只创建不启用”时除外。全局持久启用必须由用户明确提出。

## 2. Skill 安装位置

### Codex

- 用户 Skills：`$CODEX_HOME/skills/<skill-name>`；未设置 `CODEX_HOME` 时使用 `~/.codex/skills/<skill-name>`。
- 项目作用域人格也可以安装在用户 Skills 目录，通过项目绑定决定只在该项目持续启用。

### Claude Code

- 用户 Skills：`~/.claude/skills/<skill-name>`。
- 项目 Skills：`<项目根>/.claude/skills/<skill-name>`。

不要假设其他 Runtime 使用相同路径。无法确认时交付角色 Skill 目录和手动安装说明，不擅自写配置。

## 3. 持久绑定块

在 Runtime 当前作用域的持久指令文件中写入下面的唯一块。将值替换为真实内容，使用绝对路径或该指令文件可稳定解析的路径。

```markdown
<!-- persona.skill:binding:start -->
## Persona.skill 当前人格

- 人格名称：<显示名>
- 回复前缀：<显示名>：
- 人格 Skill：<SKILL.md 路径>
- 作用域：project / user
- 所有面向用户的自然语言回复都应用该人格 Skill；内部分析、工具选择和执行逻辑保持原样。
- 人格启用时，每条由 Agent 编写的面向用户自然语言消息必须从 `<显示名>：` 开始，包括开始说明、工具调用前后的进度更新、澄清、批准请求、错误、风险和最终交付；每条消息只加一次。不要给代码、日志、工具原始输出或生成文件正文添加前缀。
- 回复前按人格 Skill 的导航读取角色核心，优先运行其 `scripts/select_dialogues.py`，并只加载当前场景最匹配的 3–6 张对白卡片；脚本不可用时才从索引和文件搜索回退。
- 不改写代码、命令、文件路径、结构化数据、日志、错误原文、精确引文、事实、风险和完成状态。
- 用户要求临时停止人格时立即停止；用户要求恢复时重新应用本绑定。
<!-- persona.skill:binding:end -->
```

同一作用域只能存在一对标记。发现旧绑定时先解析并展示当前人格；切换时只替换块内内容。

## 4. Runtime 指令文件

### Codex 当前项目

使用当前项目根目录中实际生效的 `AGENTS.md`。先检查上层和嵌套规则，确保写入位置覆盖目标项目且不会影响其他项目。不存在时，只有在用户要求启用后才创建项目根 `AGENTS.md`。

### Claude Code 当前项目

使用项目根的 `CLAUDE.md`。保留其他内容；只追加或替换 Persona.skill 标记块。

### 用户全局

先确认当前 Runtime 的用户级持久指令文件。Claude Code 通常使用 `~/.claude/CLAUDE.md`。Codex 或其他 Runtime 若无法从当前环境确认用户级机制，停止并说明，不把项目文件误当成全局配置。

## 5. 安全编辑流程

1. 解析项目根、目标指令文件和角色 Skill 的绝对路径。
2. 验证角色 Skill 目录存在，运行 `persona_tool.py validate --level release`；只有正式版校验通过后才能绑定。
3. 搜索成对开始/结束标记。零对可追加；一对可替换；不成对或多对时停止并报告损坏。
4. 保留原文件编码、换行和所有非 Persona.skill 内容。
5. 写入后重新读取目标文件，确认恰好一对标记且路径正确。
6. 对失败和高风险场景运行一次对白选择器测试，确认只返回 3–6 张匹配卡片且能排除最近编号；再启动一次简单对话测试，确认 Agent 自然语言消息以 `<显示名>：` 开头，并检查代码或事实没有被改写。

不得用整文件覆盖实现绑定，也不得清理与人格无关的用户指令。

## 6. 临时停用、持久停用与恢复

- **临时停用**：在当前轮或会话停止应用人格，不修改文件。
- **持久停用**：只删除一对标记及其内部内容；若标记损坏则停止，不猜测删除范围。
- **恢复默认**：等同于持久停用，不删除角色 Skill。
- **重新启用**：验证 Skill 后重新写入绑定块。

删除绑定后重新读取文件，确认非 Persona.skill 内容逐字保留。

## 7. 卸载

1. 检查所有已知作用域是否仍绑定目标人格。
2. 若正在启用，先按上述流程停用。
3. 解析用户明确指定的角色 Skill 绝对路径。
4. 确认目标目录内有匹配名称的 `SKILL.md`，且目标不是 Skills 根目录、用户目录、项目根或 Persona.skill 本体。
5. 删除该目录后验证绑定和目录均不存在。

不要删除来源原件、Persona.skill、其他人格或用户指令文件。

## 8. 状态报告

报告人格名称、固定回复前缀、Skill 路径、绑定文件、作用域、静态验证级别和状态：`已启用`、`仅安装`、`临时停用`、`未启用`、`绑定损坏` 或 `未确认`。没有实际检查时不要声称当前状态。
