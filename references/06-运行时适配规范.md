# 运行时适配规范

## 1. v2 边界

v2 支持 16 个逻辑运行时：Codex App/CLI、Claude Code、OpenCode、WorkBuddy、CodeBuddy、Kimi Code、MiMo Code、GitHub Copilot、Gemini CLI、Cursor、Cline、TRAE、QoderWork、Deep Code、OpenClaw 和 Hermes。统一蒸馏与验证逻辑放在母 Skill 中，运行时差异只存在于用户级 Skills 路径、持久绑定能力、私有注册表、启用回执和连续状态路径。

支持分为两层：

- **完整启用型**：运行时同时公开或稳定提供用户级 Skill 目录和会话自动加载的指令文件。Persona.skill 可以安全维护唯一标记块，支持注册、启用、切换、停用和回执核验。
- **Skill-only 型**：运行时公开支持 Skill，但没有可由脚本安全维护的用户级全局人格文件。Persona.skill 支持安装、正式校验、注册、状态、连续状态和删除；角色需在会话中显式点名或由运行时按 Skill 描述调用。不得把安装或注册报告成全局启用。

“全局”只指当前 Runtime 的当前用户。各 Runtime 的私有数据与活动角色彼此独立，不静默修改其他 Runtime。MiMo Code 与 Deep Code 共享开放标准 `~/.agents/skills` 下的 Skill 文件，但注册表和连续状态仍分别保存在自己的配置根目录。

## 2. 确定路径与能力

先运行 `runtime-detect`，或显式传入 `--runtime`。自动检测只有存在唯一可靠环境或安装路径信号时才成功；有多个信号或没有信号时停止并要求选择 Runtime，不猜测。运行时 ID 接受常见别名，例如 `kimi`、`kimicode`、`mimocodex`、`copilot`、`gemini-cli`、`deep-code`。

| Runtime ID | 用户 Skills 根目录 | 用户级持久绑定文件 | 能力 |
|---|---|---|---|
| `codex` | `$CODEX_HOME/skills`，默认 `~/.codex/skills` | 非空 `$CODEX_HOME/AGENTS.override.md` 优先，否则 `AGENTS.md` | 完整启用 |
| `claude` | `$CLAUDE_CONFIG_DIR/skills`，默认 `~/.claude/skills` | `$CLAUDE_CONFIG_DIR/CLAUDE.md` | 完整启用 |
| `opencode` | `$OPENCODE_CONFIG_DIR/skills`；否则 `$XDG_CONFIG_HOME/opencode/skills` | 同根目录 `AGENTS.md` | 完整启用 |
| `workbuddy` | `~/.workbuddy/skills` | `~/.workbuddy/SOUL.md` | 完整启用 |
| `codebuddy` | `~/.codebuddy/skills` | `~/.codebuddy/CODEBUDDY.md` | 完整启用 |
| `kimi-code` | `$KIMI_CODE_HOME/skills`，默认 `~/.kimi-code/skills` | `$KIMI_CODE_HOME/AGENTS.md` | 完整启用 |
| `mimo-code` | `~/.agents/skills` | 无安全文件接口 | Skill-only |
| `github-copilot` | `~/.copilot/skills` | `~/.copilot/copilot-instructions.md` | 完整启用 |
| `gemini` | `~/.gemini/skills` | `~/.gemini/GEMINI.md` | 完整启用 |
| `cursor` | `~/.cursor/skills` | User Rules 由应用设置管理，无稳定文件接口 | Skill-only |
| `cline` | `~/.cline/skills` | `~/Documents/Cline/Rules/persona-skill.md` | 完整启用 |
| `trae` | `~/.trae/skills` 或 `~/.trae-cn/skills` | 无安全文件接口 | Skill-only |
| `qoderwork` | `~/.qoderwork/skills` | 无安全文件接口 | Skill-only |
| `deepcode` | `~/.agents/skills` | 公开 `AGENTS.md` 为项目级，无用户级文件接口 | Skill-only |
| `openclaw` | `$OPENCLAW_HOME/skills`，默认 `~/.openclaw/skills` | `$OPENCLAW_WORKSPACE/AGENTS.md`，默认 `~/.openclaw/workspace/AGENTS.md` | 完整启用 |
| `hermes` | `$HERMES_HOME/skills`，默认 `~/.hermes/skills` | `$HERMES_HOME/SOUL.md` | 完整启用 |

每个 Runtime 还接受 `PERSONA_<RUNTIME_ID>_HOME` 覆盖 Persona 工具解析到的配置根目录；它只影响 Persona.skill，不宣称对应 Agent 原生识别该环境变量。OpenClaw 工作区可用 `PERSONA_OPENCLAW_WORKSPACE` 精确覆盖。

TRAE 统一安装器默认同时安装国际版和国内版。工具从实际母 Skill 路径判断使用 `.trae` 还是 `.trae-cn`；若脱离安装副本单独调用，可用 `PERSONA_TRAE_HOME` 明确选择。测试使用临时 `--home` 时忽略真实配置目录环境变量，避免污染用户配置。

## 3. 角色目录与注册表

生成角色的稳定 ID 必须是 ASCII 小写 `persona-<slug>`，角色目录直接位于当前 Runtime 的用户 Skills 根目录且目录名与 ID 相同。显示名、别名和回复前缀允许 Unicode。

每个 Runtime 在私有 `registry.json` 中维护：

- `schema_version` 与 `active_role_id`；
- 角色稳定 ID、显示名、别名、人格类型和来源身份；
- 角色绝对路径、固定回复前缀、正式验证哈希；
- 创建和更新时间。

按显示名、别名或稳定 ID 切换时只接受精确匹配；命中多个角色才询问并列出稳定 ID。注册表 schema 损坏时停止，不自动重建覆盖。

Skill-only 运行时使用 `register`：它必须完成正式校验、写入注册表并回读角色路径与哈希，但保持 `active_role_id=null`，不生成虚假的绑定文件或启用回执。最终门禁使用 `--activation-status registered`，并重新核对注册项与角色哈希。

## 4. 唯一全局绑定

本节只适用于完整启用型运行时。绑定使用下面的唯一成对标记，由工具生成完整内容：

```markdown
<!-- persona.skill:binding:start -->
<!-- persona.skill:binding:separator-added:0 -->
...
<!-- persona.skill:binding:end -->
```

零对标记可添加，一对可原子替换；起止缺失、顺序错误、重复或缺少边界保留元数据时停止。不得猜测范围，不得覆盖整个全局指令文件。

写入流程：

1. 解析 Runtime、实际生效的全局指令文件和角色绝对路径。
2. 执行正式校验并计算角色目录哈希；失败时禁止启用。
3. 读取原文件字节，识别 UTF-8、带 BOM 的 UTF-8 或 UTF-16 及既有换行；不支持的编码停止。
4. 修改前创建带 UTC 时间戳的同目录备份；只替换标记块，原子写入。
5. 回读确认恰好一对标记、目标角色路径正确、编码和换行未改变。
6. 更新注册表并生成 `receipts/activation.json`，记录 Runtime、角色、绑定路径、角色哈希、验证哈希、绑定块哈希、整个绑定文件哈希和时间。

`completion-gate` 重新计算并比较这些值。角色文件被修改、用户在启用后修改全局指令文件、活动注册项变化、绑定块变化或回执缺失都会令启用失效；重新正式验证并执行 `enable` 后才能交付。

Skill-only 运行时调用 `enable`、`switch` 或 `disable` 时必须在任何写入前失败，并明确返回当前 Runtime 只支持 Skill 注册与显式调用。

## 5. 生命周期

- `register <角色目录>`：正式验证并注册角色，适用于 Skill-only，也可用于显式登记；不改变活动角色。
- `enable <角色目录>`：仅完整启用型；正式验证、注册、写入或替换绑定、生成回执并回读验证；重复执行幂等。
- `list`：列出当前 Runtime 的注册角色及活动 ID。
- `status`：报告持久启用能力，并检查注册表、实际绑定和回执；不把未检查状态报告为通过。
- `switch <查询>`：仅完整启用型；精确解析目标，重新正式验证，再原子替换绑定。
- `disable`：仅完整启用型；只删除 Persona.skill 成对标记、清空活动 ID 和回执，保留角色 Skill 与状态。
- `delete <查询> --yes`：若活动则先停用，再删除精确角色目录、状态和注册项。

删除前必须确认目标恰好是当前 Runtime Skills 根目录直属的 `persona-<slug>`，且与注册表绝对路径一致。永不删除 `persona` 母 Skill、Skills 根目录、用户目录、项目目录、其他角色或外部资料。注册表中即使出现外部路径或路径穿越结果也必须拒绝。

用户说“这轮别演”只在当前会话临时停止角色化，不自动改持久绑定；用户明确要求持久停用时才运行 `disable`。Skill-only 运行时没有持久活动绑定，不得报告已执行 `disable`。

## 6. 受限连续状态

每个角色在当前 Runtime 私有 `state/persona-<slug>.json` 中保存：关系摘要、情绪残留、信任、承诺、未完话题和近期表达编号。它只用于跨会话延续关系与避免重复：

- 只有这些字段发生有意义变化时才原子写入；相同状态不更新文件时间。
- 摘要与单项长度有限；承诺和未完话题最多各 12 项，近期表达编号最多 50 项，超量只保留最近项。
- 禁止 `messages`、`transcript`、`history`、`conversation` 等完整聊天记录字段。
- `reset-memory <角色>` 清空该角色连续状态，不删除角色资料、注册项或绑定。
- 连续状态不得覆盖来源事实、权限、测试结果、风险判断或用户当前明确指令，也不跨 Runtime、账号或设备同步。

## 7. 状态报告

报告 Runtime、能力层级、角色稳定 ID、显示名、角色路径、注册表路径和检查结果。完整启用型还报告实际绑定文件、回执路径和哈希核验；Skill-only 型报告 `registered` 与无持久绑定，不得显示 `enabled`。

未实际运行写 `not-run`；Runtime 程序不可用写 `runtime-unavailable`；回执或哈希不符写 `activation-stale`；注册项或角色哈希不符写 `registration-stale`。这些状态都不能算通过。
